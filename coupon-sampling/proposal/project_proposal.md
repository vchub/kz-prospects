# Digital Coupon Sampling — Проектное предложение

## 1. Суть и ценность проекта

Digital Coupon Sampling — третий сценарий семплинга, дополняющий мобильные промо-станции и
брендированные sampling zone. Пользователь получает цифровой купон через Telegram-бот и
обменивает его на охлаждённый напиток в одной из трёх точек погашения: офлайн-ретейл,
доставка или мобильная промо-точка бренда.

**Ценность для клиента (Coca-Cola):**
- Семплинг без промо-персонала, логистики и холодильного оборудования — при работе через партнёров (ретейл, доставка); на собственных промо-точках — снижение краж и контроль каждой выдачи
- Верифицированный охват: каждый купон привязан к уникальному пользователю
- Аналитика в реальном времени — корректировка кампании на ходу
- Новый digital-канал поверх офлайн-активаций, а не вместо них

**Ценность для пользователя:**
- Получает напиток там, где удобно — магазин, доставка или промо-точка
- Купон можно подарить другу
- Нет регистрации, нет приложений — только Telegram

**Ценность для ретейл-партнёра:**
- Дополнительный трафик в точку
- Сопутствующие покупки при визите за бесплатным напитком

Тестовый рынок — Казахстан. Пилот: 1–2 города.

## 2. Операционные проблемы, которые решает сценарий

| Проблема офлайн-семплинга | Решение |
|---|---|
| Охлаждённый продукт на точке | Выдача из холодильников магазина или через доставку |
| Логистика и хранение | Не нужна — продукт уже в ретейле |
| Верификация и контроль злоупотреблений | 1 купон на 1 Telegram-аккаунт, одноразовый QR-код |
| Нет данных о ходе кампании в реальном времени | Активация купона фиксируется мгновенно |

## 3. Платформа — Telegram: плюсы, минусы, альтернативы

### Плюсы

- **Верификация без OTP** — Telegram-аккаунт привязан к номеру телефона, дополнительных шагов не нужно
- **Нет отдельного приложения** — не нужно разрабатывать мобильное или веб-приложение с регистрацией и личным кабинетом; скорость запуска в разы выше
- **Купон хранится в чате** — пользователь найдёт его в истории переписки в любой момент
- **Нативный шэринг** — переслать купон другу в один клик
- **Push-уведомления** — бот присылает напоминание об истекающем купоне без отдельной инфраструктуры
- **Охват в Казахстане** — Telegram является одним из основных мессенджеров в стране, особенно среди городской аудитории

### Минусы

- **Нет Telegram — нет акции** — пользователи WhatsApp или без смартфона исключены, нет fallback
- **Ограниченные данные о пользователе** — Telegram передаёт только ID, имя и username; нет email для CRM-интеграции
- **Интерактивность ограничена** — купон реализуется как брендированное изображение в чате (с логотипом, визуалом напитка, QR-кодом), но без анимации, эффектов и геймификации — это возможно только в Telegram Web App
- **Зависимость от платформы** — любые изменения в политике Telegram влияют на проект

### Варианты реализации внутри Telegram

Проект может быть реализован в двух форматах — они не исключают друг друга, а представляют два этапа.

#### Вариант 1 — Telegram-бот (рекомендуется для пилота)

Купон отображается как брендированное изображение в чате бота.

| Плюсы | Минусы |
|---|---|
| Быстрый запуск — минимум разработки | Купон — статичная картинка, нет анимации и интерактива |
| Купон хранится в истории чата, пользователь всегда найдёт | Нет геймификации — scratch, shake, spin wheel недоступны |
| Нативный шэринг — переслать сообщение в один клик | — |
| Push-уведомления без отдельной инфраструктуры | — |

#### Вариант 2 — Telegram Web App (TWA, этап 2)

Полноэкранное веб-приложение, открывающееся внутри Telegram по кнопке в боте.

| Плюсы | Минусы |
|---|---|
| Полноэкранный branded experience — анимация, фирменные цвета | Требует отдельной веб-разработки (React / Vue); больше времени и бюджета |
| Интерактивный купон: scratch, shake, spin wheel | Купон не живёт в истории чата — пользователь возвращается через кнопку в боте |
| Геймификация реализуется нативно | Относительно новый API Telegram — возможны баги на разных устройствах |
| Больше данных о поведении пользователя внутри экрана | Два продукта вместо одного: бот + веб-приложение, сложнее поддерживать |

**Рекомендация:** пилот — Telegram-бот. При подтверждении конверсии и интересе клиента к усилению брендинга — переход на TWA на этапе 2.

### Альтернативы вне Telegram

| Вариант | Плюсы | Минусы |
|---|---|---|
| **Промо-лендинг + SMS** | Охватывает всех, нет зависимости от мессенджера | Нужна верификация по OTP, отдельная разработка, купон не хранится нативно |
| **WhatsApp Business API** | Большой охват | Дорогой API, сложная модерация в регионе |
| **QR на промо-материалах → лендинг** | Работает без мессенджера | Нет верификации пользователя, сложнее контролировать злоупотребления |

## 4. Точки погашения купона

Купон действует в трёх каналах:

**1. Офлайн-ретейл**
Пользователь приходит в Magnum или CU, активирует купон в боте, показывает QR-код на экране кассиру. Кассир сканирует — напиток бесплатен.

**2. Доставка**
Пользователь активирует ваучер в боте, выбирает Glovo и получает промо-код. Затем оформляет заказ в Glovo и вводит промо-код в соответствующее поле — стоимость напитка списывается. Промо-код не QR, а текстовый — вводится вручную.

**3. Мобильная промо-точка бренда**
На брендированной промо-станции (sampling zone) промоутер предлагает пользователю получить купон через бот прямо на месте — по QR-коду или ссылке на стойке. Пользователь сканирует, получает купон и тут же активирует его у промоутера. Промоутер подтверждает активацию через отдельный интерфейс (веб-панель или промо-бот) и выдаёт напиток.

Этот канал решает задачу учёта выдач на промо-точках — каждый образец привязан к верифицированному пользователю, исключая повторное получение и ручной счёт.

```

```
Пользователь на промо-точке
        ↓
Промоутер предлагает QR / ссылку → пользователь открывает бот
        ↓

Сценарий 3 полностью решает проблему учёта выдач на промо-станциях: нет ручного счёта, нет повторных получений, данные поступают онлайн — в отличие от ретейл-каналов с лагом.

## 6. Механика купона

В системе существуют два разных объекта — **ваучер** и **код погашения**. Это принципиальное разделение.

### Ваучер — что пользователь получает при выдаче

Ваучер — это **право на напиток**. Он не привязан к конкретному партнёру и не содержит партнёрского кода. Пользователь получает брендированную карточку:

- Визуал бренда (цвета, картинка напитка, логотип)
- Текст «1 бесплатный напиток»
- Внутренний идентификатор (`FT-4827`) — наш, не партнёрский
- Срок действия
- Кнопка активации

Ваучер можно шэрить — QR на карточке ведёт обратно в бот:
```
t.me/FreeTasteBot?start=coupon_FT-4827
```
Друг сканирует → попадает в бот → забирает ваучер → активирует когда готов.

### Код погашения — что генерируется в момент активации

Когда пользователь нажимает **«Активировать»** и выбирает партнёра — бот берёт код из пула этого партнёра и показывает его в виде QR-кода. Именно этот код сканирует кассир или промоутер.

Источники кодов по партнёрам:

| Партнёр | Источник кода | Валидация |
|---|---|---|
| Magnum | Пул кодов от Magnum (передаётся до старта кампании) | Сканер кассы Magnum |
| CU | Пул кодов от CU | Сканер кассы CU |
| Glovo | Промо-код от Glovo | Поле в приложении Glovo |
| Промо-точка | Генерируем сами | Наша веб-панель промоутера |

Пулы кодов от ретейл-партнёров — обязательный пункт партнёрского соглашения до старта кампании.

### Шэринг

Пользователь шэрит ваучер (право на напиток), а не код партнёра. Реферальная ссылка одноразовая: напиток достаётся первому, кто откроет ссылку. После передачи исходный пользователь **теряет право активации** — ваучер полностью переходит к другу. Друг входит в бот, видит ваучер и может активировать его самостоятельно. Он также видит кнопку «Получить свой купон» — и становится новым участником акции.

## 7. Почему активация — отдельный шаг

POS-системы ретейл-партнёров передают данные **с задержкой и пакетно** — онлайн-интеграция с кассами на пилоте нереалистична.

Активация в боте — это **событие реального времени** в момент, когда пользователь стоит у кассы. Это позволяет:

- Видеть живую статистику на дашборде без ожидания POS-данных
- Сверять активации с POS-данными постфактум
- Фиксировать пики активности по времени и географии

```
Купон выдан → Купон активирован → Сканирование на кассе
     ↑                ↑                      ↑
наша система   реальное время        POS-данные (с лагом)
```

На промо-точках активация подтверждается онлайн через панель промоутера — здесь лага нет.

## 8. Контроль злоупотреблений

- **1 купон на 1 Telegram-аккаунт** за кампанию — по Telegram ID
- **QR-код генерируется только в момент активации** — не при получении ваучера
- **Одноразовое использование** — после сканирования код деактивируется мгновенно
- **TTL кода — 15 минут** — после активации QR-код действителен ограниченное время, что исключает повторное использование скриншота
- **Реферальная ссылка одноразовая** — ваучер переходит к другу; после передачи исходный пользователь теряет право активации
- **Промо-точка**: промоутер подтверждает выдачу через авторизованный интерфейс, что исключает выдачу без сканирования

## 10. Аналитика и дашборд

| Событие | Источник | Задержка |
|---|---|---|
| `coupon_issued` | Telegram-бот | Реальное время |
| `coupon_activated` | Telegram-бот | Реальное время |
| `promo_point_confirmed` | Панель промоутера | Реальное время |
| `pos_confirmed` | POS ретейл-партнёра | Лаг (пакетная передача) |

Дашборд в реальном времени:
- Выдано / активировано купонов (воронка)
- Популярность брендов: Cola / Fanta / Sprite
- Распределение по каналам: ретейл / доставка / промо-точка
- Пики активности по времени суток и дням
- Конверсия: выдан → активирован → подтверждён

## 11. Технический стек и реализация

### Компоненты

| Компонент | Решение | Обоснование |
|---|---|---|
| Telegram-бот | Python + aiogram 3 | Зрелый фреймворк, быстрая разработка |
| Backend API | FastAPI (Python) | Единый язык со стеком бота, высокая производительность |
| База данных | PostgreSQL | Надёжное хранение купонов, транзакции при активации |
| Кэш / сессии | Redis | Хранение состояния диалога бота, rate limiting |
| Генерация QR-кода | qrcode (Python) | QR-формат — читается камерами телефонов и современными сканерами на кассах |
| Дашборд | Metabase (open source) + PostgreSQL | Быстрый старт без кастомной разработки |
| Панель промоутера | Веб-приложение с двумя режимами | Сканер камерой + ручной ввод кода — промоутер выбирает по ситуации |
| Хостинг | VPS (Hetzner / DigitalOcean) | Достаточно для пилота, низкая стоимость |

### Ключевые технические решения

**QR-код совместим с ретейл-сканерами**
QR-формат читается современными сканерами на кассах Magnum и CU без доработки оборудования. Код генерируется в момент активации и действителен **15 минут** — защита от повторного использования скриншота. Пользователь предупреждается об этом в боте в момент активации.

**Защита от скриншота**
Активированный QR-код имеет ограниченный TTL (15 минут). Кассир должен сканировать его онлайн — бот мгновенно помечает код как использованный. Повторное сканирование вернёт ошибку.

**Панель промоутера**
Веб-страница по защищённой ссылке с двумя режимами работы:
- **Сканер камерой** — промоутер наводит камеру телефона на QR пользователя
- **Ручной ввод кода** — промоутер вводит код (`FT-4827-XKQM`) с экрана пользователя

Промоутер выбирает режим по ситуации: сканер удобен в помещении, ввод вручную надёжнее на улице при бликах. После подтверждения — зелёный экран с названием напитка, красный — если код недействителен или уже использован.

**Интеграция с ретейлом**
На пилоте не требует API-интеграции с POS: ретейл-партнёр получает выгрузку активированных кодов (CSV) для ежедневной сверки. Полноценная POS-интеграция — следующий этап.

**Масштаб пилота**
Монолитная архитектура на одном VPS достаточна для нагрузки пилота. Декомпозиция на микросервисы — при масштабировании.

## 12. Что не охвачено / открытые вопросы

- **Язык интерфейса** — нужен русский + казахский; бот должен определять язык по настройкам Telegram
- **Нет Telegram** — пользователи без аккаунта выпадают из воронки; для пилота принимаем это ограничение, на следующем этапе — лендинг как резервный канал
- **Данные пользователей** — Telegram передаёт минимум (ID, имя); нет email для CRM. Если клиенту нужна база — рассмотреть опциональный запрос контакта в боте
- **Как ретейл сверяет данные** — нужно договориться о формате и частоте выгрузки активаций (CSV ежедневно / webhook)
- **Офлайн-режим промоутера** — что если на промо-точке нет интернета? Нужен fallback: бумажный список заранее сгенерированных кодов
- **Юридическая сторона** — правила акции, обработка персональных данных (Telegram ID — персональные данные по законодательству РК), согласие пользователя
- **Бюджет на продукт** — кто несёт стоимость напитков: клиент компенсирует ретейлу или включает в маркетинговый бюджет? Нужно согласовать с трейд-командой

## 13. KPI пилота

**Marketing:**
- Количество выданных купонов
- Конверсия в активацию: целевой показатель **15–20%**
- Стоимость контакта и стоимость активации

**Retail:**
- Дополнительный трафик в партнёрские точки
- Сопутствующие покупки (средний чек)

**Operational:**
- Доля валидных активаций (без злоупотреблений)
- Точность сопоставления активаций с POS-данными

## 15. Масштабирование

**Этап 1 — Пилот** (1–2 города)
Telegram-бот, 2 ретейл-партнёра (Magnum + CU), 1 служба доставки (Glovo), промо-точки бренда. Тест механики и конверсии.

Дополнительные ретейл-сети и delivery. Расширение географии. Telegram Web App для усиления брендинга.

Масштабирование на всю страну. POS-интеграция с ключевыми партнёрами. Интеграция в регулярные маркетинговые кампании.
