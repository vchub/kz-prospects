# Digital Coupon Sampling — Клиентское предложение

## Кратко

**Проблема.** Офлайн-семплинг охлаждённого напитка дорог в операционном исполнении: нужны логистика, холодильники, промо-персонал и ручной контроль выдач. При этом данные о ходе кампании либо приходят с лагом, либо не собираются вовсе.

**Решение.** Пользователь получает ваучер в Telegram-боте и активирует его только в момент получения напитка. После активации система выдаёт партнёрский код погашения: для retail это QR-код на кассу, для доставки это промо-код, для промо-точки это наш собственный одноразовый код. За счёт этого кампания получает живую аналитику без необходимости в онлайн-интеграции с POS на пилоте.

**Что это даёт Coca-Cola.**
- Семплинг без логистики и холодильного оборудования при работе через retail и delivery-партнёров
- Контроль каждой выдачи на собственных промо-точках
- Верифицированный охват: один участник, один ваучер, одноразовая активация
- Аналитику в реальном времени по выдачам и активациям
- Новый digital-канал поверх офлайн-активаций, а не вместо них

**Почему это важно.** Ключевая инновация не в “купоне в Telegram” как таковом, а в разделении на два этапа:

1. **Ваучер** как маркетинговое право на напиток
2. **Активация** как операционное событие в момент получения

Именно это разделение позволяет видеть ход кампании в реальном времени, даже если retail-партнёр передаёт POS-данные позже.

## Какие задачи решает сценарий

Бриф ставит три практические задачи. Эта механика закрывает все три:

| Задача | Как решается |
|---|---|
| Охлаждённый продукт | Выдача идёт из холодильников магазина или через delivery-партнёра |
| Логистика и хранение | Продукт уже находится в retail-канале, везти его на точку не нужно |
| Контроль злоупотреблений | Один ваучер на один Telegram-аккаунт, активация одноразовая, код действует ограниченное время |

Дополнительный эффект: бренд видит активации в ходе кампании и может корректировать размещение и нагрузку по каналам, не дожидаясь пост-анализа.

---

## Почему Telegram

Для пилота Telegram даёт лучший баланс между скоростью запуска и пользовательским трением.

- не нужно отдельное приложение
- не нужна регистрация через OTP
- купон хранится в истории чата
- бот может сам напомнить об истечении ваучера
- ссылку на ваучер можно переслать другу

Ограничение тоже надо назвать прямо: этот сценарий работает только для аудитории с Telegram. Для пилота это допустимо; если канал покажет конверсию, на следующем этапе можно добавить резервный entry point через лендинг.

## Как работает механика

### 1. Пользователь получает ваучер

Пользователь видит рекламу, UGC, 2GIS-размещение или QR-код на промо-точке, переходит в Telegram-бот и выбирает напиток.

Бот выдаёт **ваучер**:

- брендированную карточку
- внутренний ID
- срок действия
- кнопку `Активировать`

На этом этапе партнёрского кода ещё нет. Это важно: ваучер можно хранить, пересылать и активировать позже.

### 2. Пользователь активирует ваучер

Когда пользователь готов получить напиток, он нажимает `Активировать` и выбирает канал:

- `Magnum / CU`
- `Glovo`
- `Промо-точка`

Только в этот момент система выдаёт **код погашения**:

- для retail: QR-код из пула партнёра
- для delivery: текстовый промо-код из пула партнёра
- для промо-точки: наш собственный одноразовый код

Именно активация является событием реального времени, которое попадает в аналитику сразу.

### 3. Пользователь получает напиток

- в retail: показывает QR-код кассиру
- в delivery: вводит промо-код в заказе
- на промо-точке: показывает код промоутеру, который подтверждает выдачу через веб-панель

После использования код становится недействительным.

---

## Три канала погашения

### Офлайн-ретейл

Пользователь приходит в Magnum или CU, активирует ваучер в боте и получает QR-код. Кассир сканирует код, напиток выдаётся из магазина.

Для бренда это означает:

- не нужно везти продукт на отдельную точку
- не нужны холодильники на месте
- можно использовать существующий retail stock

### Доставка

Пользователь активирует ваучер, выбирает Glovo и получает текстовый промо-код. Затем оформляет заказ и вводит код в приложении.

Для бренда это означает:

- получение продукта без визита в офлайн-точку
- дополнительный канал охвата
- тот же принцип real-time activation без отдельной сложной интеграции на пилоте

### Промо-точка бренда

Пользователь сканирует QR на стойке, получает ваучер и сразу активирует его на месте. Промоутер подтверждает выдачу через веб-панель.

Для бренда это означает:

- контроль каждой выдачи
- исключение ручного счёта
- снижение риска повторных получений и злоупотреблений
- онлайн-данные по промо-точкам без лага

## Почему активация вынесена в отдельный шаг

Это центральный элемент всей системы.

Если retail-партнёр передаёт подтверждение с кассы с задержкой, бренд всё равно видит:

- сколько ваучеров выдано
- сколько ваучеров активировано
- в каком канале и когда возник пик спроса

То есть логика выглядит так:

`Выдача ваучера -> Активация -> Подтверждение партнёра`

Где:

- выдача и активация видны сразу
- подтверждение продажи может прийти позже

На пилоте это снимает необходимость строить полноценную live POS-интеграцию до старта.

---

## Аналитика, которую получает бренд

В реальном времени можно видеть:

- выданные ваучеры
- активированные ваучеры
- популярность брендов: Cola / Fanta / Sprite
- распределение по каналам: retail / delivery / promo-point
- пики активности по времени
- конверсию `выдан -> активирован`

Позже можно сверять эти данные с POS-выгрузкой retail-партнёра.

Целевой ориентир пилота: конверсия `выдан -> активирован` на уровне **15–20%**.

Важно: это не гарантированный результат, а реалистичный ориентир пилота, который опирается на смежные digital sampling кейсы.

---

## Почему сценарий выглядит жизнеспособным

Внешние кейсы подтверждают не “полное совпадение модели”, а отдельные рабочие элементы:

- **Coca-Cola #TakeATaste Now** подтверждает механику `digital contact -> voucher -> retail redemption`
- **Sampl / Bio&Me** подтверждает, что digital sampling с retail redemption может давать заметную конверсию
- **Telegram retail promo кейсы** подтверждают, что Telegram может быть рабочим интерфейсом для массовой промо-механики

Корректная интерпретация здесь такая: рынок уже доказал жизнеспособность компонентов модели. Наш проект объединяет эти компоненты в один pilot-ready сценарий для Казахстана.

## Риски и как мы их закрываем

| Риск | Как закрывается |
|---|---|
| Часть аудитории не использует Telegram | Для пилота принимается как ограничение канала; при подтверждении спроса можно добавить лендинг как резервный вход |
| Retail-партнёр задерживает пул кодов | Формат и объём пулов фиксируются до старта кампании в партнёрском соглашении |
| Пользователь пытается переиспользовать код | Код генерируется только при активации и действует ограниченное время |
| Фейковые или множественные участия | Ограничение один ваучер на один Telegram-аккаунт, rate limiting, одноразовое погашение |
| На промо-точке нестабильный интернет | Для промо-точки нужен резервный операционный сценарий с заранее подготовленными кодами |
| POS-данные приходят с лагом | Аналитика строится на событиях выдачи и активации, а POS используется для последующей сверки |

## Экономика пилота

Сценарий убирает или резко снижает расходы на:

- логистику продукта на отдельные sampling-точки
- холодильное оборудование
- часть промо-персонала при работе через retail и delivery

Основные статьи бюджета:

- разработка и запуск Telegram-бота
- backend и учёт механики активации
- панель промоутера
- аналитический дашборд
- digital media
- компенсация стоимости продукта retail-партнёрам

## Пилот и масштабирование

### Этап 1. Пилот

- 1–2 города
- 2 retail-партнёра
- 1 delivery-партнёр
- собственные промо-точки бренда

Цель этапа:

- проверить механику
- проверить конверсию
- оценить операционные ограничения по партнёрам

### Этап 2

- расширение по географии
- подключение новых retail и delivery партнёров
- усиление брендированного опыта

### Этап 3

- масштабирование на всю страну
- углубление интеграций с ключевыми партнёрами
- использование механики в регулярных кампаниях и product launch сценариях

---

## Итог

Digital Coupon Sampling решает не только маркетинговую, но и операционную задачу.

Он позволяет:

- выдавать бесплатный напиток без тяжёлой офлайн-логистики
- контролировать каждую активацию
- получать реальные данные по кампании в ходе её работы
- использовать retail, delivery и brand promo-points в одной системе

Для пилота это выглядит как прагматичный и реалистичный запуск: достаточно простой по инфраструктуре, но достаточно сильный по измеримости и управлению.
