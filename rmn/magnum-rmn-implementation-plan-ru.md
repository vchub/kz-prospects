# Magnum RMN (Проект Alpha) - План реализации

Область: только офлайн/«фиджитал» RMN (цифровой e-commerce инвентарь контролируется Kaspi). План предполагает, что Magnum владеет in-store экранами, «умными» весами и приложением Magnum Club, и фокусируется на измеримой in-store медиа с привязкой к loyalty‑данным.

## 1) Цели и метрики успеха
- Монетизация физического трафика магазинов через измеримые медиа‑пакеты для FMCG‑поставщиков.
- Построение data и ad‑serving слоя, независимого от Kaspi.
- Доказуемая closed‑loop атрибуция для офлайн‑продаж.

Основные KPI
- Медиа‑выручка на магазин в месяц
- Инкрементальный рост продаж продвигаемых SKU (тест vs контроль)
- Fill rate и CPM/CPA‑таргеты
- Доля opt‑in и размер активной аудитории
- Время запуска кампании (готовность self‑service)

## 2) Поэтапный план выполнения

Фаза 0 - Выравнивание и управление (4-6 недель)
- Определить RMN‑юнит с отдельным P&L, команда: продукт, данные, инженерия, продажи.
- Юр. проверка: что можно делать с данными клиента в офлайн RMN без нарушения JV‑ограничений.
- Шорт‑лист вендоров для signage CMS, ad server и CDP (build vs buy).

Фаза 1 - Основа данных (8-12 недель)
- Создать единый shopper ID для офлайн‑данных (loyalty‑карта + хеш телефона).
- Стандартизировать ID каталога товаров между POS, инвентарем и ценами.
- Построить минимальный CDP: сегменты по истории покупок и гео магазина.
- Запустить privacy‑слой: управление согласиями и минимизация данных.

Фаза 2 - Активация in-store инвентаря (10-14 недель)
- Обновить «умные» весы и LED‑экраны до адресуемых endpoint‑ов.
- Развернуть легкий ad decisioning service (сначала rules‑based, затем ML).
- Начать с 10-20 пилотных магазинов в Алматы/Астане; измерить lift.

Фаза 3 - Loyalty + мобильные триггеры (8-12 недель, параллельно Фазе 2)
- Добавить in-store геозоны в приложении (только opt‑in) для push‑офферов.
- Запустить «basket‑aware» триггеры на кассе (купон‑коды).
- Валидировать closed‑loop атрибуцию через loyalty ID.

Фаза 4 - Self‑Service портал (12-16 недель)
- Портал для поставщиков: бронирование показов, загрузка креативов.
- Базовые дашборды: показы, охват, lift продаж, ROAS.
- Модель ценообразования: CPM для экранов, CPA/CPS для таргет‑офферов.

Фаза 5 - Масштабирование (постоянно)
- Расширение до 100+ магазинов и дополнительных форматов.
- Динамическая оптимизация креатива и сезонные триггеры.
- Off‑site активация только если JV‑ограничения позволят.

## 3) Архитектура ПО (высокоуровнево)

Ключевые системы
- Data ingestion: POS, loyalty, inventory и pricing фиды.
- CDP/Segment store: сегменты и профили покупателей.
- Ad decisioning: rules engine, затем ML.
- Content delivery: signage CMS и device management.
- Measurement: атрибуция и отчетность.

Референс‑архитектура
1) Слой ingestion данных
- POS‑события (почти real‑time)
- Loyalty‑события (приложение + сканы карт)
- Product catalog (SKU master)
- Store metadata (локация, формат)

2) Identity и CDP
- Shopper ID сервис (хеш телефона, маппинг loyalty ID)
- Segment builder (RFM, category affinity, basket patterns)
- Consent registry (флаги opt‑in)

3) Ad serving и правила
- Campaign rules engine
- Inventory scheduler (экраны, тайм‑слоты)
- Frequency capping на покупателя

4) Каналы доставки
- In-store экраны (LED, shelf‑edge, весы)
- Push‑уведомления в приложении
- Опционально: landing page через in‑store Wi‑Fi

5) Measurement
- Контрольные vs экспонированные когорты
- Lift продаж по SKU
- ROAS и окна атрибуции

Рекомендуемый стек (agnostic)
- Event streaming: Kafka/PubSub
- Storage: Postgres для метаданных, ClickHouse/BigQuery для событий
- API‑слой: GraphQL/REST
- Ad decisioning: кастомный rules engine + Redis cache
- Dashboard: Metabase/Looker

## 4) Доступ к 3rd‑party API и интеграциям

Обязательные интеграции
- Signage CMS API (инвентарь устройств, расписание контента)
- Push‑notification сервис (Firebase/APNS)
- POS интеграция (Magnum internal или vendor API)
- Loyalty platform API (Magnum Club)
- Inventory и catalog система (Symphony RetailAI или аналог)

Опциональные интеграции
- Платежные/кошельки для redemption (если разрешено)
- Внешний партнер для измерения brand lift
- Geofencing SDK (если приложение поддерживает локации)

Работы по доступу к API
- Определить владение данными и права доступа в JV‑договорах.
- Создать vendor‑neutral API gateway для избегания lock‑in.
- Запустить sandbox для поставщиков (тест креативов и отчетности).

## 5) Маркетинговая стратегия (плюсы и минусы)

Позиционирование
- «Офлайн RMN с детерминированной атрибуцией» для FMCG‑брендов.
- «Фиджитал‑конверсия» для поставщиков, которым нужен in‑store lift.

Плюсы
- Эксклюзивный доступ к данным о in‑store конверсии (offline closed‑loop).
- Высокий intent в точке покупки.
- Низкая конкуренция относительно Kaspi digital inventory.
- Премиальные размещения (касса, весы, shelf‑edge).

Минусы
- Высокий hardware CapEx и обслуживание на масштабе.
- Более медленный запуск кампаний по сравнению с digital‑платформами.
- Нужна сильная sales‑enablement функция для переноса trade marketing бюджетов.
- Ограниченный охват вне магазина.

План запуска маркетинга
- Пилотные кейсы с 2-3 топ FMCG‑брендами.
- Bundle‑цены как замена части trade marketing fees.
- «Гарантированные» офферы (минимум показов или rebate).
- Совместный PR с поставщиками для подтверждения измеримого lift.

## 6) Реестр рисков и меры
- Ограничения JV по данным: фокус на офлайн‑активах; юр. проверка на старте.
- Операционная сложность: начинать с ограниченного числа магазинов.
- Низкая adoption у рекламодателей: совместная разработка пакетов с ключевыми поставщиками.
- Качество данных: нормализация SKU и store ID.

## 7) Deliverables
- Устав RMN бизнес‑юнита
- MVP CDP + определения сегментов
- План пилотного запуска в магазинах
- MVP supplier‑портала и дашбордов
- Первые 3 бренд‑кейса с анализом lift
